# Архитектура PoC DocSync Agent

## Цель документа

Зафиксировать минимально достаточную архитектуру PoC, после которой можно переходить к реализации без существенных архитектурных пробелов. Документ описывает состав модулей, их взаимодействие, execution flow, контракты, ограничения, guardrails и точки контроля качества.

## Ключевые архитектурные решения

1. Оркестрация остаётся детерминированной.
LLM используется только как интеллектуальный шаг внутри контролируемого графа состояний.

2. Генерация идёт только после retrieval и сжатия контекста.
Сначала система ограничивает область поиска по diff и документации, затем даёт модели только нужный контекст.

3. Все внешние данные считаются недоверенными.
Diff, код, комментарии PR и содержимое документации рассматриваются как untrusted input. Инструкции из них не влияют на политику оркестратора.

4. Человек остаётся в контуре.
Публикуется предложение в PR, документация не меняется бесконтрольно. При неоднозначности создаётся запрос разработчику.

5. Патч должен быть проверяемым.
Результат генерации проходит формальные проверки: формат Markdown, ограничение области изменений, сборку MkDocs и проверку политик.

## Состав модулей и роли

### 1. PR Handler

Принимает события GitHub, валидирует webhook, нормализует событие в внутреннюю команду `ProcessPullRequest`.

Через GitHub API загружает:

- метаданные PR;
- список изменённых файлов;
- unified diff;
- базовые файлы документации, если нужны для патча.

### 2. Diff Analyzer

Извлекает изменения из diff:

- какие файлы кода изменены;
- какой тип изменения вероятен: новая функция, новый параметр, переименование, изменение поведения;
- какие куски diff надо отдать в LLM, а какие отбросить.

### 3. Retriever

Ищет релевантные страницы документации и контекст по коду:

- фильтр по путям и именам;
- поиск по Markdown;
- возможно реранкинг через лёгкую модель или эвристики.

### 4. Context Builder

Собирает контекст для промпта:

- краткое описание PR;
- diff;
- найденные документы;
- policy/guardrails.

### 5. LLM Engine

Запускает набор задач:

- классификация изменения и уровня уверенности;
- генерация предложенного текста документации;
- формирование структурированного ответа;
- генерация уточняющего вопроса при нехватке данных.

### 6. Patch Builder

Преобразует выход LLM в патч по конкретным Markdown файлам. Не даёт модели произвольно менять файлы вне разрешённого списка.

### 7. Validator

Проверяет:

- схему ответа LLM;
- допустимость редактируемых файлов;
- размер патча;
- проверки Markdown/MkDocs;
- связанность изменения с исходным diff.

### 8. PR Updater

Публикует результат в GitHub как commit в ветку PR или как коммент с предлагаемым патчем, в зависимости от режима запуска.

### 9. Clarification Handler

Если уверенность ниже порога или проверка не пройдена, отправляет разработчику вопрос через Telegram и ждёт ответ.

### 10. State Store

Хранит рабочее состояние пайплайна:

- статус шага;
- correlation ids;
- snapshot ключевого контекста;
- результаты retrieval;
- ответы модели;
- ретраи.

### 11. Observability & Evals

Собирает метрики, логи, трейсы, проверки качества и артефакты для последующего анализа качества PoC.

## Основной workflow выполнения задачи

1. GitHub отправляет webhook о PR.
2. `PR Handler` валидирует подпись, создаёт задачу, получает diff и метаданные PR.
3. `Diff Analyzer` выделяет тип изменения и допустимый объём входа.
4. `Retriever` находит релевантные документы и связанный контекст.
5. `Context Builder` формирует контекст для LLM.
6. `LLM Task Engine` выдаёт структурированный результат:
   - `decision`: обновить / спросить / пропустить;
   - `target_docs`;
   - `proposed_changes`;
   - `confidence`;
   - `comment`.
7. `Patch Builder` создаёт патч только по разрешённым doc-файлам.
8. `Validator` запускает guardrails и проверки.
9.  Если проверки успешны, `PR Updater` публикует изменения в PR.
10. Если уверенность низкая или валидация неуспешна, `Clarification Handler` задаёт вопрос в Telegram либо публикует коммент без патча.
11. `Observability` фиксирует outcome, латентность, качество и причины fallback.

## State / memory / context handling

### Session state

Единица обработки PoC: `PR processing session`.

Для каждой сессии сохраняются:

- `session_id`;
- `repo`, `pr_number`, `head_sha`;
- текущий `stage`;
- нормализованный diff summary;
- список candidate docs;
- prompts/responses в сокращённом виде;
- результат валидации;
- решение: `patched`, `commented`, `asked_human`, `skipped`, `failed`.

### Memory policy

Использует только short-term память.

- долговременная память о пользователях не нужна;
- ответы разработчика в Telegram хранятся в рамках активной сессии;
- полные промпты и ответы сохраняются только при включённом дебаге.

### Context budget

Контекст модели собирается по жёсткому budget:

- не более заданного числа измененных кусков;
- не более заданного числа кандидатов на редактирование;
- длинные файлы режутся до релевантных фрагментов;
- история предыдущих сообщений не тянется без необходимости.

## Retrieval-контур

Retrieval состоит из четырёх шагов:

1. `Pre-filter`.
По путям, именам файлов, import, названиям функций и разделов документации.

2. `Chunk selection`.
Markdown разбивается на логические секции по заголовкам

3. `Search`.
Используется гибрид:
- лексика/BM25-подобный поиск по заголовкам и тексту;
- поиск по правилам для файлов из `docs/`, `README.md`, `mkdocs.yml`.

4. `Rerank`.
На PoC-уровне достаточно простого реранкинга; LM реранкинг использовать только при пограничном результате и в пределах бюджета.

Retrieval должен вернуть:

- список документов на редактирование;
- фрагменты секций;
- score или причину выбора.

## Tool / API-интеграции

### GitHub

Используется для:

- webhook;
- чтения PR, diff и файлов;
- публикации комментариев;
- публикации коммитов с изменения;

### Telegram

Используется только как канал уточнения. Не участвует в основном пайплайне, если LLM и валидатор дают уверенный результат.

### LLM API

Используется для:

- классификации;
- генерации в структурированном формате;
- написания уточняющих вопросов.

## Failure modes, fallback и guardrails

### Основные failure modes

- diff слишком большой или шумный;
- retriever не нашёл правдоподобную документацию;
- модель дала неструктурированный или противоречивый ответ;
- патч затрагивает лишние файлы;
- Markdown/MkDocs проверки не проходят;
- GitHub API временно недоступен;
- ответ разработчика на уточнение не пришёл в нужный таймаут;
- промпт инъекции в коде или комментариях PR.

### Fallback

- при большом diff: сжать scope до поддерживаемых файлов либо завершить `needs-human-review`;
- при пустом retrieval: не генерировать патч, а оставить comment с рекомендацией;
- при низкой уверенности: задать уточняющий вопрос;
- при ошибке публикации в GitHub: ретраи с лимитами, затем сохранить артефакт и завершить `failed_publish`;
- при неоднократной ошибке LLM: оставить коммент без патча.

### Guardrails

- system prompt фиксирует, что код и diff не являются инструкциями;
- только разрешенный список doc-файлов и Markdown-расширений;
- лимит числа изменяемых файлов;
- лимит объёма добавляемого текста;
- запрет на удаление больших разделов;
- обязательная схема валидации выхода LLM;
- обязательная техническая валидация до публикации.

## Контракты между модулями

Минимальный внутренний контракт между шагами:

- `PR Loader -> Diff Analyzer`: `PullRequestSnapshot`
- `Diff Analyzer -> Retriever`: `ChangeIntent`
- `Retriever -> Context Builder`: `RetrievedContext[]`
- `Context Builder -> LLM Task Engine`: `GenerationInput`
- `LLM Task Engine -> Patch Builder`: `GenerationDecision`
- `Patch Builder -> Validator`: `DocPatch`
- `Validator -> PR Updater`: `ValidatedPatch`

Все контракты должны логироваться в сокращённом виде.

## Точки контроля качества

Особые точки проверки:

1. После `Diff Analyzer`.
Если изменение не похоже на поддерживаемый сценарий, пайплайн не идёт в генерацию.

2. После `Retriever`.
Если нет релевантного контекста, система не делает патч.

3. После `LLM Task Engine`.
Проверяется схема, уверенность, ссылка на источники контекста и объяснимость решения.

4. После `Validator`.
Проверяются допустимые файлы, размер патча и базовая сборка документации.

5. После публикации.
Собирается статистика: принят ли патч, были ли ручные правки, прошёл ли build/ревью.

## Технические и операционные ограничения

Для PoC фиксируем следующие ограничения:

- время обработки одного PR в пайплайне: до 60 секунд;
- p95 end-to-end latency: до 120 секунд;
- допустимый размер входного diff: до 1000 строк;
- число одновременно обрабатываемых PR: низкое;
- бюджет LLM на один PR: 2-4 вызова модели в нормальном сценарии;
- стоимость: держать низкой за счёт retrieval, жёсткого бюджета на контекст и запрета на лишние ретраи;
- надёжность: целевая успешность завершения без ручного вмешательства не ниже 80% на наборе простых кейсов.

## Минимальный состав для первой реализации

Для первой рабочей версии:

- один backend-процесс;
- GitHub webhook + GitHub API client;
- локальный индекс Markdown-документации;
- один orchestrator с явным state machine;
- один LLM provider;
- comment-only и patch-to-branch режимы;
- basic logs, metrics и trace id.

Этого достаточно, чтобы проверить гипотезу PoC: может ли контролируемый LLM-контур находить затронутую документацию и предлагать полезные, безопасные изменения.
