# Спецификация модуля Observability / Evals

## Цель

Сделать поведение PoC измеримым: понимать, когда система помогает, когда ошибается и где именно ломается.

## Метрики

Базовые технические метрики:

- end-to-end latency;
- длительность шагов orchestrator;
- число LLM вызовов на PR;
- token usage и approximate cost;
- success/failure rate по интеграциям;
- retry count;
- fallback rate.

Качественные метрики:

- доля PR, где найден релевантный doc candidate;
- доля валидных предложенных патчей;
- доля принятых человеком изменений;
- доля случаев, где потребовалось уточнение;
- build pass rate для Markdown/MkDocs проверок.

## Логи

Логируются:

- `session_id`, `repo`, `pr_number`, `head_sha`;
- входной тип события;
- коды причин на каждом переходе;
- найденные доки из retrieval;
- confidence и outcome;
- причина fallback или failure.

Не логируются в полном виде:

- секреты;
- избыточные фрагменты репозитория;
- персональные данные.

## Трейсы

Желательно иметь один trace на одну сессию на PR:

- ingest
- GitHub read
- retrieval
- LLM generation
- validation
- publish

## Evals

Минимальный eval для PoC:

- добавление параметра функции;
- переимнование функции;
- добавление нового модуля;
- изменение CLI-команды;
- изменение поведения без изменения сигнатуры;
- случай с устаревшей документацией;
- случай с prompt injection в комментарии или коде.

Для каждого кейса фиксируются:

- ожидался ли патчи;
- какие доки ожидались;
- ожидаемый резлуьтат;
- должен ли был быть fallback.

## Точки контроля

- проверка логов ошибок;
- ревью топ-ошибок LLM схемы или качества;
- контроль дрифта по acceptance rate после смены модели или промпта.
