# Спецификация модуля Retriever

## Цель

Найти минимальный, но достаточный контекст документации для генерации безопасного патча.

## Вход

- `ChangeIntent`
- список изменённых файлов
- diff summary
- path hints и symbol hints

## Источники

- `README.md`
- `docs/**/*.md`
- `mkdocs.yml` при наличии
- ограниченно: кодовые комментарии и названия сущностей из изменённых файлов

## Индекс

- индексируются Markdown-файлы;
- единица индексации: секция по заголовку;
- для каждой секции сохраняются `path`, `heading`, `tokens_estimate`, `last_modified`.

## Поиск

1. Path/rule filtering.
2. Лексический поиск по названиям файлов, заголовкам, названиям.
3. Эвристики:
   - совпадение имени функции или модуля;
   - близость пути;
   - наличие слов `usage`, `configuration`, `api`, `parameters`.

## Реранкинг

В PoC по умолчанию heuristic-only:

- top-k кандидатов = 3-5;
- реранкинг по сумме lexical score, path affinity и section density.

LM-реранкинг допустим только как дополнительный шаг при слабом результате и при наличии бюджета.

## Выход

`RetrievedContext[]`:

- `doc_path`
- `section_title`
- `excerpt`
- `score`
- `selection_reason`

## Ограничения

- не индексировать бинарные файлы;
- не включать в контекст больше 3-5 документов;
- не включать слишком длинные секции целиком без обрезки;
- если нет правдоподобного совпадения, вернуть пустой результат, а не слабый шумный контекст.

## Обработка ошибок
- индекс пустой: отправить запрос на ребилд
- поиск пустой: вернуть `no_relevant_docs`;
- diff слишком общий: снизить score и инициировать human fallback.
