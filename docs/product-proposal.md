# Product Proposal

## Обоснование идеи

Документация проектов часто устаревает быстрее, чем код.

Основные причины:

- разработчики не обновляют документацию при изменениях
- в процессе ревью фокус смещён на код
- документация разбросана по нескольким файлам

В результате:

- пользователи получают неверные инструкции
- API используется неправильно
- новые участники проекта тратят больше времени на изучение системы

Цель DocSync Agent — автоматически обнаруживать несоответствия между кодом и документацией и предлагать обновления. Сервис интегрируется в обработку PR и обновляет документацию вместе с кодом. Это позволяет сделать документацию синхронизированной с кодовой базой.

## Цель проекта

Создать PoC агентного сервиса, который автоматически предлагает изменения документации при изменении кода.

## Метрики успеха

### Продуктовые метрики

- количество принятых изменений документации
- снижение времени на обновление документации

### Агентские метрики

- точность определения затронутых страниц документации
- доля случаев, когда агенту требуется уточнение от разработчика
- количество корректных предложений документации

### Технические метрики

- успешность генерации патча документации
- процент успешных сборок документации (MkDocs)
- время обработки одного PR

## Потенциальные сценарии использования

### Сценарий 1 — изменение API

Разработчик добавляет новый параметр функции.

Агент:
1. обнаруживает изменение сигнатуры функции
2. находит соответствующий раздел документации
3. добавляет описание нового параметра

### Сценарий 2 — добавление нового модуля

Разработчик добавляет новый модуль в проекте.

Агент:

1. обнаруживает изменение
2. добавляет раздел о модуле в документацию
3. описывает модуль, его функции и примеры работы

### Сценарий 3 — уточнение поведения

Если агент не уверен в изменениях:

1. агент формирует вопрос
2. отправляет его разработчику через Telegram
3. использует ответ для обновления документации

## Edge cases

- переименование функций: документация может ссылаться на старое имя
- изменение поведения без изменения сигнатуры: сложно автоматически понять изменение
- документация уже устарела: агент может предложить некорректные изменения

## Ограничения

### Технические ограничения

Ожидаемые параметры системы:

- время обработки PR: до 60 секунд
- p95 latency: до 120 секунд
- размер diff: до 1000 строк

### Операционные ограничения

- использование API GitHub
- ограничение бюджета LLM запросов
- ограниченное количество репозиториев

## Архитектурный набросок

Основные модули системы:

- PR Listener
  - отслеживает новые Pull Request через GitHub webhook
- Diff Analyzer
  - анализирует изменения в коде
- Documentation Retriever
  - находит релевантные страницы документации
- Documentation Generator
  - генерирует предложения по обновлению документации
- PR Updater
  - добавляет изменения документации в Pull Request
- Developer Communication Module
  - отправляет вопросы разработчику через Telegram

## Потенциальный data flow

1. GitHub webhook сообщает о новом PR
2. система получает diff изменений
3. Diff Analyzer извлекает изменения
4. Retriever находит соответствующие страницы документации
5. Documentation Generator генерирует обновления документации
6. PR Updater добавляет изменения в Pull Request

## Что делегируется LLM

LLM используется для:

- анализа изменений diff
- поиска соответствующих разделов документации
- генерации обновлений Markdown

## Что не делегируется LLM

LLM не используется для:

- управления GitHub API
- выполнения коммитов
- запуска сборки документации
